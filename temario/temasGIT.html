Volver al Curso

1. ¿Qué es Git?

Git es una Máquina del Tiempo para tu código.

Permite guardar "fotos" (commits) de tu proyecto en diferentes momentos. Si rompes algo, puedes volver al pasado en segundos. Es esencial para trabajar en equipo sin sobreescribir el trabajo de otros.

2. Configuración Inicial

Antes de empezar, dile a Git quién eres (esto queda en el historial). 

$ git config --global user.name "Tu Nombre"
$ git config --global user.email "tu@email.com"
Esto se hace una sola vez en tu computadora.

Evaluación 1: Conceptos

¿Dónde se guarda el historial de tu proyecto?

En la nube de Google.
En la carpeta oculta .git de tu proyecto.
En la RAM de tu PC.

3. Init & Clone

Dos formas de empezar:

1. Desde cero: 

cd mi-proyecto
git init
2. Descargar existente: 
git clone https://github.com/usuario/proyecto.git

4. El Flujo de Git (Workflow)

Imagina que vas de compras (Shopping Cart Analogy):

  1. Working Directory: Estás en el pasillo eligiendo productos (Editando archivos).
  2. Staging Area (git add): Pones los productos en el carrito. Estás listo para pagar.
  3. Repository (git commit): Pagas y te dan el recibo. La compra (cambio) es oficial.

5. Comandos Básicos

# Ver qué archivos cambiaste
git status

# Agregar todo al carrito
git add .

# Guardar la foto con mensaje
git commit -m "Agregué el navbar"

Evaluación 2: Comandos

¿Qué hace `git add .`?

Guarda los cambios permanentemente.
Prepara todos los archivos modificados para el próximo commit.
Sube el código a internet.

6. Ramas (Branches)

El Multiverso de tu código.
`git branch experimento` crea una copia paralela. Puedes borrar todo en la rama "experimento" y tu rama "main" seguirá intacta y segura.

7. Merge (Fusión)

Cuando tu experimento fue un éxito, lo integras. 

# Voy a la rama principal
git checkout main

# Traigo los cambios de experimento
git merge experimento

8. Conflictos

Pasan cuando dos personas tocan la misma línea del mismo archivo.
Git no sabe cuál elegir, así que te dice: "¡Ayuda! Resuélvelo tú". Tienes que editar el archivo, decidir el código final, y hacer commit de nuevo.

Evaluación 3: Ramas

¿Es buena práctica trabajar todo directamente en la rama 'main'?

No, es arriesgado. Deberías usar ramas (features) para cada tarea.
Sí, es más rápido.

9. Git vs GitHub

Git: Lo usas en tu terminal (Local).
GitHub/GitLab: La red social del código (Remoto). Guardan copia de tus repositorios para trabajar en equipo.

10. Pull Requests (PR)

En equipos, no haces merge directo a Main.
Abres una Pull Request: "Hola equipo, cambié el botón. ¿Pueden revisar mi código?".
Si aprueban, se fusiona. Si encuentran errores, los corriges antes.

11. .gitignore

El club VIP de archivos ignorados.
Aquí pones `node_modules`, archivos de configuración con contraseñas (`.env`), etc. Git fingirá que no existen.

Evaluación Final

¿Qué comando envías tus commits locales a GitHub?

git upload
git push
git send
